# escape.id 高速チケット購入 Chrome 拡張 仕様書（ドラフト v0.1）

- 作成日: 2026-03-01
- 対象: `https://escape.id/` のチケット購入フロー
- 目的: 指定時刻に最短レイテンシで購入操作を実行し、取りこぼしを最小化する

## 1. ゴール

1. 指定した日時（JST）に購入操作を自動実行できること
2. 手動操作より早く、かつ失敗時に即時リトライできること
3. ユーザーのセッション情報・個人情報を安全に扱うこと

## 2. スコープ

### 対象

- Chrome 拡張（Manifest V3）
- ログイン済み状態での購入ページ自動操作
- 事前入力（枚数、券種、支払い選択など）の自動化
- 指定時刻トリガー、リトライ、結果通知

### 非対象

- CAPTCHA 回避
- 不正なアクセス制限回避（レート制限回避、認証突破など）
- サーバー侵入やAPIの不正利用

## 3. 前提・制約

1. ユーザーが事前に `escape.id` へログイン済みであること
2. 利用規約に反しない範囲で運用すること
3. DOM構造の変更に追随できるよう、セレクタを設定可能にすること
4. 拡張はローカル実行のみ（外部サーバー必須にしない）

## 4. 主要ユースケース

1. ユーザーがイベントURLと購入条件を事前登録する
2. 発売時刻の数分前に対象ページを開き、入力可能項目を事前セットする
3. 指定時刻に購入ボタン押下フローを自動実行する
4. 失敗時はエラー原因に応じて一定回数リトライする
5. 成否を通知（ポップアップ + ログ）する

## 5. 機能要件

## 5.1 設定機能（Options Page）

- 必須設定
  - 対象イベントURL
  - 発売日時（JST、秒・ミリ秒指定）
  - 券種・枚数
  - リトライ上限回数
- 任意設定
  - 実行前ウォームアップ秒数（例: 120秒）
  - セレクタ上書き設定（DOM変更対策）
  - クリック間隔（ms）
  - タイムアウト（各ステップ）

## 5.2 時刻同期

- `Date.now()` のみでなく、外部時刻API（HEADレスポンス Date 等）でドリフト推定
- ドリフト補正値を保持し、トリガー時刻を補正
- 失敗時はローカル時刻でフォールバック

## 5.3 購入フロー自動化（Content Script）

1. 対象ページ読込完了を検出
2. 必要フィールドの存在確認
3. 枚数・券種・同意チェック等を自動入力
4. 発売時刻まで待機（高精度タイマー）
5. トリガー時刻到達で購入アクション実行
6. 確認画面遷移時は次ボタンを連鎖実行
7. 完了画面/エラー画面を判定して結果記録

## 5.4 リトライ戦略

- 対象エラー
  - 一時的在庫切れ表示
  - 遷移失敗/ボタン非活性
  - 一時的ネットワークエラー
- 方針
  - 最大 `N` 回まで短間隔リトライ（例: 200ms, 400ms, 800ms）
  - 致命的エラー（認証切れ、決済失敗など）は即停止

## 5.5 通知・ログ

- 実行ログ（時刻、ステップ、結果、エラー種別）を `chrome.storage.local` に保存
- 成功/失敗をChrome通知またはポップアップで表示
- ワンクリックで直近ログをコピー可能

## 6. 非機能要件

1. トリガー精度: 目標 ±30ms 以内（ローカル環境依存）
2. 入力・クリック処理: DOM準備済み時は 100ms 以内に開始
3. エラー時の復帰: 1秒以内に次アクションへ移行
4. 個人情報保護: 機微データは必要最小限のみ保存

## 7. セキュリティ要件

1. `host_permissions` は `https://escape.id/*` のみに限定
2. 外部送信はデフォルト無効（テレメトリなし）
3. 決済情報そのものは保存しない
4. 設定エクスポート時はマスキング対応

## 8. 拡張構成（案）

- `manifest.json` (MV3)
- `service_worker.js`
  - スケジューリング
  - タブ監視
  - 通知
- `content_script.js`
  - DOM検出
  - 自動入力/クリック
  - 状態遷移管理
- `options.html` / `options.js`
  - 設定UI
- `popup.html` / `popup.js`
  - 状態確認・手動実行・ログ表示

## 9. 状態遷移（簡易）

1. `IDLE`
2. `WARMUP`
3. `READY`
4. `TRIGGERED`
5. `PROCESSING`
6. `SUCCESS` or `FAILED`

## 10. 失敗パターンと対処

1. ログイン切れ: 即停止し再ログイン要求
2. DOM変更: セレクタ設定のフォールバック順で再試行
3. 時刻ずれ: 次回実行前に自動再同期
4. タブクラッシュ/リロード: Service Worker から自動復旧

## 11. テスト要件

1. 単体テスト
  - 設定検証
  - 時刻補正計算
  - リトライ制御
2. 結合テスト
  - 疑似DOMで購入フローを再現
  - 遅延/エラー注入で復帰確認
3. 実運用リハーサル
  - 本番直前にダミーイベントでE2E確認

## 12. 実装フェーズ

1. Phase 1: 設定UI + 手動実行自動入力
2. Phase 2: 指定時刻トリガー + ログ
3. Phase 3: リトライ + セレクタ上書き + 通知
4. Phase 4: テスト強化 + 運用手順書作成

## 13. 受け入れ基準（MVP）

1. 指定時刻に自動実行できる
2. 主要画面遷移で自動購入完了まで到達できる
3. 失敗時に理由がログで判別できる
4. 設定変更が再起動後も保持される
